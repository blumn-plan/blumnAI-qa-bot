# blumnAI-qa-bot

정책문서 기반 QA 챗봇 — 사내에서 팀별로 깔아 쓰는 모듈.

> ⚠️ **이 레포는 private 입니다.** 사내 자산이므로 외부 공유 금지.

## 📌 한 줄 요약

> **정책·설계서 기반의 AI 소통을 통해, '문의 → 협의 → 문서 수정 → 배포' 에 이르는 복잡한 제품 개선 프로세스를 [클릭 몇 번] 으로 단축하는 지능형 협업 시스템.**

QA/운영자가 정책문서·화면설계서에 대해 질문하면 AI 가 즉시 답변하고, 정책 변경이 필요하면 기획자에게 합의 요청으로 자동 전달됩니다. 본인 PC 의 Claude Max 구독을 활용해 **추가 API 비용 0원 (C 모드)** 또는 유료 API (A 모드) 로 운영 가능.

원본은 헤이데어 기획팀의 `heythere_planer/qa` 에서 fork — 같은 회사 다른 팀도 쉽게 가져다 쓰도록 분리·모듈화한 버전.

---

## 🚀 왜 이 봇이 필요한가 — 핵심 장점 4가지

### 1. 커뮤니케이션 비용의 극적 절감 (Zero-Waste)

- **논리적 설득 자동화** — QA·개발자와 기획자 간의 불필요한 말싸움이나 긴 설명이 사라짐. AI 가 변경 전/후, 근거, 대상 파일 위치까지 정제된 논리로 기획자를 알아서 설득
- **구구절절한 지라(Jira) 작성 생략** — 현상과 개선안을 인간이 다시 문서로 포맷팅해 등록할 필요 없이, 시스템이 명확한 변경 요청 사항을 `qa/decisions/` 에 markdown 으로 자동 저장 → 기획자에게 즉시 전달

### 2. 기획자의 리소스 파괴적 혁신 (Prompt-Driven Update)

- **'복사–붙여넣기–배포' 프로세스** — 기획자가 문서를 직접 열어 텍스트를 고치고 화면설계서를 수정할 필요 X. AI 가 만들어 준 **최적의 프롬프트를 복사해 Claude Code(Antigravity)에 붙여넣기**만 하면 정책문서 · AI 답변 룰 · 화면설계서가 한 번에 자동 업데이트
- **빠른 이슈 파악** — 기획자는 상단 요약본만 보고 이슈를 3초 만에 파악한 뒤 배포까지 끝냄 (기획자 UI 는 `qa-planner.html`)

### 3. AI-to-AI 소통을 통한 명확성 극대화

사람이 개입하면서 발생하는 의사소통의 왜곡 (컨텍스트 오해 · 휴먼 에러) 이 없습니다. 질문자의 의도가 답변 AI → 기획단 AI (Claude Code 프롬프트) 로 다이렉트 전달되므로, 가장 정확하고 명확한 형태로 제품이 수정됩니다.

### 4. 개발 레이어까지의 확장성

기획단에서 끝나는 것이 아니라, 업데이트된 내용이 개발자에게도 자동 전달됩니다. 개발자 역시 해당 내용을 프롬프트 형태로 복사해 코드나 개발 환경에 바로 적용 가능 — '기획-개발' 싱크가 완벽히 일치.

## 💡 이렇게 활용하세요 — 실전 시나리오 8가지

| # | 시나리오 | 질문 예시 | 봇의 대응 |
|---|---|---|---|
| 1 | **정책문서에서 찾기 힘들 때** | "결제 실패 시 재시도 정책 어디?" | 관련 §번호를 찾아 답변. 인용된 `§X-Y` 는 클릭하면 해당 절로 자동 스크롤 |
| 2 | **화면과 정책이 다를 때** | 화면 캡처 첨부 + "정책과 실제 화면이 다른데 어느 쪽이 맞아?" | drift 감지 후 판단 + [📤 기획자에게 전달] 유도. 기획자가 patch bump 진행 |
| 3 | **특정 액션·상태의 화면 확인** | "반복 횟수가 무한일 때 메시지관리>리스트 발송회차 컬럼 어떻게 표시돼?" 이미지 보여줘 | 정책·storyboard 안에 있는 관련 화면 이미지를 답변에 인용해 시각적으로 답변 |
| 4 | **화면만 있고 관련 정책 모를 때** | 화면 캡처만 붙이고 "이 화면 관련 정책 뭐야?" | vision 으로 화면 이해 → 그 화면에 해당하는 정책 안내 |
| 5 | **정책 내용 이해가 안 갈 때** | "이 §3-2 무슨 뜻이야?" → "그럼 이 케이스일 때는?" | 채팅 히스토리 유지 → 반복 문의 OK. 사람은 귀찮아할 수 있는 후속 질문도 무제한 |
| 6 | **원하는 정책이 없을 때** | "이 케이스에 대한 정책이 없는데 어떻게 정해야 할까?" | 봇이 초안 제안 → [📤 기획자에게 전달] 로 신설 요청 자동 접수 |
| 7 | **정책 버전 이력 추적** | "대시보드_v0.1.5 → v0.1.6 사이에 뭐가 바뀌었어?" | 버전 이력 표와 📍 변경 위치 빠른 색인을 활용해 요약 답변 |
| 8 | **엣지 케이스 도출** | "이 정책에서 놓친 예외 상황 있을까?" | AI 창의성 활용해 검토 안 된 케이스 제시 → 기획자 전달로 정책 보완 |

### 부수 효과

- **신입 온보딩** — 신입 QA·개발자가 정책을 통째로 정독하지 않고 자유 질문으로 파악
- **답변 스타일 팀 학습** — [📝 개선 요청] 이 `qa/feedback/` 에 누적 → 다음 답변부터 자동 반영 → 팀 커뮤니케이션 톤으로 봇이 진화

## ⚖️ 한계 · 알아두면 좋은 것

- **컨텍스트는 "현재 선택한 문서 1개"** — 봇은 사용자가 좌측에서 선택한 정책·화면설계서 하나를 근거로 답변. 여러 정책을 동시에 비교하는 질문은 잘 못함 (사용자가 정책을 하나씩 물어 보는 걸 권장)
- **자동 코드 검증은 제한적** — 기본 봇은 정책 markdown 만 fetch. 실제 admin/frontend 코드까지 자동 비교하려면 확장 개발 필요 (헤이데어팀 원본에는 이 로직 있음)
- **정책에 없으면 봇도 모름** — 정책이 부실하면 봇 답변도 부실. 초기 세팅 후 [📝 개선 요청] / [📤 기획자에게 전달] 축적으로 점진적 개선
