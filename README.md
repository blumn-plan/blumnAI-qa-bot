# blumnAI-qa-bot

정책문서 기반 QA 챗봇 — 사내에서 팀별로 깔아 쓰는 모듈.

> ⚠️ **이 레포는 private 입니다.** 사내 자산이므로 외부 공유 금지.

## 📌 한 줄 요약

> **정책·설계서 기반의 AI 소통을 통해, '문의 → 협의 → 문서 수정 → 배포' 에 이르는 복잡한 제품 개선 프로세스를 [클릭 몇 번] 으로 단축하는 지능형 협업 시스템.**

QA/운영자가 정책문서·화면설계서에 대해 질문하면 AI 가 즉시 답변하고, 정책 변경이 필요하면 기획자에게 합의 요청으로 자동 전달됩니다.

**기본 운영 모드는 A 모드 (Anthropic API 사용, 유료).** 설치가 단순하고 PC 상시 가동이 필요 없어서 일반 팀에 권장하는 셋업입니다. 별도로 본인 PC 의 Claude Max 구독을 우회 활용해 **추가 API 비용 0원 (C 모드)** 으로도 운영 가능하지만, PC 상시 가동·터널 유지 등의 부담이 있어 특수 상황용입니다.

## 🏛 이 레포는 뭐고, 각 팀은 어떻게 쓰나

**이 레포 (`blumn-plan/blumnAI-qa-bot`)** = 기획팀이 지속적으로 유지·업데이트하는 **재사용 가능한 코어 봇**. 이 레포 자체는 어느 팀 정책 md 도 갖고 있지 않아요 — 그건 각 팀 정책 레포 몫.

각 서비스팀은 이 코어를 **자기 정책 레포 안 `.blumnAI-qa-bot/` 폴더에 통째로 복사해서** 씁니다 — 👇 **직접 복사할 필요 X.** 아래 🟢 세팅하기 프롬프트가 자동으로 처리합니다:

```
[코어 (이 레포)]                    [팀 정책 레포]
blumn-plan/blumnAI-qa-bot   →→→→   blumn/○○-planer
     (봇 파일 통째)                   ├ blumnAI-qa-bot.config.yml   ← 팀 설정 (경로·비번·모델 등)
                                     ├ .blumnAI-qa-bot/             ← 코어 사본 (봇 실행 파일)
                                     ├ (팀 정책 md - 원래 있던 것)
                                     └ qa/                          ← 봇이 자동 생성 (결정·피드백)
```

**세팅하기 프롬프트가 자동으로 처리** — 사용자가 손댈 것 없음:
- 코어에서 팀 운영에 필요한 런타임 파일만 골라 `.blumnAI-qa-bot/` 아래 배치 (코어 유지보수용 `docs/`, `examples/`, `.github/workflows/` 등은 제외)
- `blumnAI-qa-bot.config.yml` 을 팀 레포 루트에 생성 + 팀 정보 자동 채움
- 팀 기존 정책 md 파일들은 **절대 안 건드림** (경로 이동 없이 그대로 참조)

**3가지 원칙만 기억**:

1. ❌ **fork 하지 마세요** — fork 방식은 코어 업데이트 받기 어려움. 파일 복사 방식이 정답. 복사는 🟢 세팅하기 프롬프트가 자동
2. 🔔 **코어에 새 버전 나오면 자동 알림** — 각 팀의 봇 화면 상단에 배너 자동 표시 + (설정 시) 기획팀즈 채널로 웹훅 알림. 알림 오면 🟠 업데이트하기 프롬프트 한 번으로 사본 갱신 (팀 자체 정책 파일은 절대 안 건드림)
3. 🔄 **각 팀 자체 정책 변경** — 사용자가 봇에 질문하고 [📤 기획자에게 전달] 하면 AI 가 정책 md 를 자동 patch (아래 🔁 표준 협업 사이클). 코어 업데이트와 완전히 별개
4. 🎁 **코어에 개선을 반영하고 싶으면** — 팀 사본에서 먼저 수정·검증한 뒤, `blumn-plan/blumnAI-qa-bot` 에 PR 오픈. 기획팀 리뷰 후 머지 → 자동 알림으로 모든 팀에 배포. 팀 사본의 수정이 원본에 자동으로 흘러가지는 **X**, 반드시 PR + 리뷰. 상세 절차는 [CONTRIBUTING.md](CONTRIBUTING.md)

자세한 코어-사본 원칙은 [docs/05-UPGRADE.md §핵심 원칙](docs/05-UPGRADE.md#핵심-원칙--무엇이-덮어쓰이고-무엇이-보존되나) 참고.

## 🚀 왜 이 봇이 필요한가 — 핵심 장점 4가지

### 1. 커뮤니케이션 비용의 극적 절감 (Zero-Waste)

- **논리적 설득 자동화** — QA·개발자와 기획자 간의 불필요한 말싸움이나 긴 설명이 사라짐. AI 가 변경 전/후, 근거, 대상 파일 위치까지 정제된 논리로 기획자를 알아서 설득
- **구구절절한 지라(Jira) 작성 생략** — 현상과 개선안을 인간이 다시 문서로 포맷팅해 등록할 필요 없이, 시스템이 명확한 변경 요청 사항을 `qa/decisions/` 에 markdown 으로 자동 저장 → 기획자에게 즉시 전달

### 2. 기획자의 리소스 파괴적 혁신 (Prompt-Driven Update)

- **'복사–붙여넣기–배포' 프로세스** — 기획자가 문서를 직접 열어 텍스트를 고치고 화면설계서를 수정할 필요 X. AI 가 만들어 준 **최적의 프롬프트를 복사해 Claude Code(Antigravity)에 붙여넣기**만 하면 정책문서 · AI 답변 룰 · 화면설계서가 한 번에 자동 업데이트
- **빠른 이슈 파악** — 기획자는 상단 요약본만 보고 이슈를 3초 만에 파악한 뒤 배포까지 끝냄 (기획자 UI 는 `qa-planner.html`)

### 3. AI-to-AI 소통을 통한 명확성 극대화

사람이 개입하면서 발생하는 의사소통의 왜곡 (컨텍스트 오해 · 휴먼 에러) 이 없습니다. 질문자의 의도가 답변 AI → 기획단 AI (Claude Code 프롬프트) 로 다이렉트 전달되므로, 가장 정확하고 명확한 형태로 제품이 수정됩니다.

### 4. 개발 레이어까지의 확장성

기획단에서 끝나는 것이 아니라, 업데이트된 내용이 개발자에게도 자동 전달됩니다. 개발자 역시 해당 내용을 프롬프트 형태로 복사해 코드나 개발 환경에 바로 적용 가능 — '기획-개발' 싱크가 완벽히 일치.

## 💡 이렇게 활용하세요 — 실전 시나리오 8가지

| # | 시나리오 | 질문 예시 | 봇의 대응 |
|---|---|---|---|
| 1 | **정책문서에서 찾기 힘들 때** | "결제 실패 시 재시도 정책 어디?" | 관련 §번호를 찾아 답변. 인용된 `§X-Y` 는 클릭하면 해당 절로 자동 스크롤 |
| 2 | **화면과 정책이 다를 때** | 화면 캡처 첨부 + "정책과 실제 화면이 다른데 어느 쪽이 맞아?" | drift 감지 후 판단 + [📤 기획자에게 전달] 유도. 기획자가 patch bump 진행 |
| 3 | **특정 액션·상태의 화면 확인** | "반복 횟수가 무한일 때 메시지관리>리스트 발송회차 컬럼 어떻게 표시돼?" 이미지 보여줘 | 정책·storyboard 안에 있는 관련 화면 이미지를 답변에 인용해 시각적으로 답변 |
| 4 | **화면만 있고 관련 정책 모를 때** | 화면 캡처만 붙이고 "이 화면 관련 정책 뭐야?" | vision 으로 화면 이해 → 그 화면에 해당하는 정책 안내 |
| 5 | **정책 내용 이해가 안 갈 때** | "이 §3-2 무슨 뜻이야?" → "그럼 이 케이스일 때는?" | 채팅 히스토리 유지 → 반복 문의 OK. 사람은 귀찮아할 수 있는 후속 질문도 무제한 |
| 6 | **원하는 정책이 없을 때** | "이 케이스에 대한 정책이 없는데 어떻게 정해야 할까?" | 봇이 초안 제안 → [📤 기획자에게 전달] 로 신설 요청 자동 접수 |
| 7 | **정책 버전 이력 추적** | "대시보드_v0.1.5 → v0.1.6 사이에 뭐가 바뀌었어?" | 버전 이력 표와 📍 변경 위치 빠른 색인을 활용해 요약 답변 |
| 8 | **엣지 케이스 도출** | "이 정책에서 놓친 예외 상황 있을까?" | AI 창의성 활용해 검토 안 된 케이스 제시 → 기획자 전달로 정책 보완 |

### 부수 효과

- **신입 온보딩** — 신입 QA·개발자가 정책을 통째로 정독하지 않고 자유 질문으로 파악
- **답변 스타일 팀 학습** — [📝 개선 요청] 이 `qa/feedback/` 에 누적 → 다음 답변부터 자동 반영 → 팀 커뮤니케이션 톤으로 봇이 진화

## ⚖️ 한계 · 알아두면 좋은 것

- **답변 근거는 "현재 선택한 문서 1개" + 프로젝트 문서 카탈로그** — 봇은 선택 문서 본문을 우선 근거로 하되, 프로젝트 안 다른 문서 목록은 항상 인지하고 있어서 *"이 케이스는 캠페인_만들기 §2-1 에 있으니 좌측에서 선택하세요"* 처럼 관련 문서 안내 가능. 여러 정책 본문을 진짜로 종합하고 싶으면 `bot.include_all_docs: true` (config) 로 전체 문서 인젝션 모드 활성 (20문서 이하 팀 권장, Anthropic prompt caching 사용)
- **코드 검증은 옵션 기능** — 기본 봇은 정책 markdown 만 fetch. 추가로 `code_repo` 를 컨피그에 연결하면 봇이 GitHub 코드 검색 결과를 함께 참고해서 *"정책은 X 인데 실제 코드는 Y 다"* 같은 drift 판정까지 가능. 셋업은 [docs/06-CONNECT-CODE.md](docs/06-CONNECT-CODE.md) 참고 (프롬프트 한 개로 끝남)
- **정책에 없으면 봇도 모름** — 정책이 부실하면 봇 답변도 부실. 초기 세팅 후 [📝 개선 요청] / [📤 기획자에게 전달] 축적으로 점진적 개선

## 🚦 어디부터 보세요 — 상황별 "프롬프트 하나로 시작"

이 봇은 **비개발자 기획자·QA 도 Claude Code 옆에 두고 설치·연결·테스트까지 가능**하도록 설계됐어요. 아래 상황 중 하나를 골라서 **회색 박스 프롬프트를 그대로 Claude Code (또는 Antigravity) 에 붙여넣으면** AI 가 관련 가이드를 알아서 읽고 스텝별로 안내합니다.

### 언제 어떤 프롬프트를 쓰나 — 시간 순서

```
[처음 셋업]                🟢 세팅하기 (필수) — 설치 + 연결 + 첫 테스트 자동 진행
       ↓
[팀에 URL 공유]            📢 실제로 매일 사용 시작 (04-OPERATE 안내 참고)
       ↓
[몇 주~몇 달 운영]          ...
       ↓
[봇 답이 이상해지면]        🔵 점검하기 (필요할 때만) — 정책 md 진단·정비
[신규 팀원 온보딩]          🟣 알려주기 (필요할 때만) — 사용법 개인 교습
[코어 업그레이드 알림]      🟠 업데이트하기 (알림 왔을 때만) — 6개월-1년에 한 번
```

**🟢 세팅하기 만 필수** 입니다. 나머지는 해당 상황이 실제로 발생했을 때만 꺼내 쓰는 도구예요. 순서대로 다 할 필요 X.

> 🔒 API key / GitHub PAT 같은 **시크릿 값은 절대 채팅창에 붙여넣지 마세요.** AI 가 명령어를 실행해줄 뿐, 시크릿 입력은 터미널 프롬프트 (`Enter a secret value:`) 에 👤 본인이 직접.

   실제로 봇 창 열어서 아래 순서로 해보라고 하나씩 지시:
   ① 좌측에서 정책 md 하나 선택
   ② 우측에 예시 질문 던지기 (07-FIRST-TEST 시나리오 1)
   ③ 답변 받고 [↻ 새 대화] · [📝 개선 요청] · [📤 기획자에게 전달]
      3가지 버튼이 언제 어떤 케이스에서 쓰이는지 각각 설명
   ④ 실제로 [📝 개선 요청] 눌러서 답변 톤 개선 요청 하나 넣어보기

   ⑤ 실제로 [📤 기획자에게 전달] 눌러서 합의문 하나 생성해보기
      → 좌측 사이드바 📤 전달이력 에 대기 상태로 뜨는지 확인

4. (기획자 역할이면) 기획자 시점 시뮬레이션 —
   qa-planner 접속 후:
   ① 방금 협업자가 만든 대기 항목 선택
   ② 합의문 본문 검토
   ③ [📋 §B-2 프롬프트 복사] 클릭 → Antigravity 창에 붙여넣기 →
     정책 md 가 실제로 자동 patch 되는 과정 지켜보기
   ④ [✅ 적용함으로 표시] 로 종결
   ⑤ (대비 학습) [🚫 보류로 표시] 도 한 번 눌러보고 사유 남기기 →
     협업자 창으로 돌아가서 초록 배너로 사유 노출되는지 확인

5. 마지막 요약 —
   "지금 배운 흐름 3줄로 요약: 질문→답변→(만족/개선요청/기획자전달) →
