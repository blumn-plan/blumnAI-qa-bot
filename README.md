## ⚡ 30초 요약

1. QA·개발자가 정책·화면에 대해 질문 → 봇이 **관련 §번호와 화면 캡처를 인용해서** 즉시 답변
2. 답이 이상하면 [📝 답변 규칙]으로 톤 조정, **정책을 변경해야 하면** [📤 기획전달]
3. 전달된 요청은 기획자 창에서 검토 → **[프롬프트 복사]** 눌러 클로드코드에 붙여넣으면 정책 md 와 화면설계서가 자동으로 수정됩니다
4. **개발자 없이도 운영 가능** — 기획자와 QA 만 있으면 셋업부터 운영까지 클로드코드가 대신 처리

**기본 모드는 A 모드 (Anthropic API 사용, 유료)** — 설치가 단순하고 PC 상시 가동이 필요 없습니다.  
C 모드 (Max 구독 우회, PC 상시 가동 필요) 는 특수 상황용이며 [부록](docs/03-CONNECT-BOT.md#부록--c-모드-pc-max-우회-특수-상황용)에 정리해뒀습니다.

## 🏛 이 레포는 뭐고 어떻게 쓰나

### 📖 1. 이해하기 — 이 레포가 뭔가

**이 레포 (`blumn-plan/blumnAI-qa-bot`) = 봇의 원본** 입니다. 기획팀이 지속적으로 유지·업데이트하는 **재사용 가능한 원본 봇** 이에요. 이 레포 안에는 어떤 팀의 정책 md 도 들어있지 않아요 — 그건 각 팀 정책 레포의 몫.

> **용어 정의** — 이 문서에서 아래 세 표현은 모두 **같은 뜻** 이에요:  
> **"원본"** = **"원본 봇"** = **"코어 (Core)"** = 이 레포 (`blumn-plan/blumnAI-qa-bot`)  
>  
> 개발 문서에서는 "코어" 라는 영어 유래 용어를 쓰기도 하지만, 이 README 는 "원본" 으로 통일했어요. 다른 문서·주석에서 "코어" 를 만나면 같은 뜻으로 이해하시면 됩니다.

---

### 🔧 2. 활용하기 — 각 팀이 어떻게 쓰나

각 서비스팀은 이 원본 봇을 **자기 정책 레포 안의 `.blumnAI-qa-bot/` 폴더에 통째로 복사해서** 씁니다.

아래는 관계를 이해하기 위한 **참고 그림** 이에요 (복사하지 않아도 됩니다):

```
📖 아래 박스는 그림 · 붙여넣기용 X · 이해용으로만 훑어보세요.

[원본 봇 (이 레포)]                  [팀 정책 레포]
blumn-plan/blumnAI-qa-bot   →→→→   blumn/○○-planer
     (봇 파일 통째)                   ├ blumnAI-qa-bot.config.yml   ← 팀 설정
                                     ├ .blumnAI-qa-bot/             ← 원본 봇 사본
                                     ├ (팀 정책 md — 원래 있던 것)
                                     └ qa/                          ← 봇이 자동 생성
```

> 📥 **실제로 복사·붙여넣기 할 프롬프트는 아래** [🟢 세팅하기 — 처음 깔 때 (필수)](#-세팅하기--처음-깔-때-필수) 섹션 안에 있어요. 그 안 회색 박스만 통째로 클로드코드에 붙여넣으면 자동 설치.

### ✅ 3. 기억할 4가지 원칙

1. ❌ **fork 하지 마세요** — 파일 복사 방식이 정답입니다. fork 는 원본 봇 업데이트를 받기 어려워져요. **복사는 아래 [🟢 세팅하기](#-세팅하기--처음-깔-때-필수) 섹션에 있는 프롬프트에 포함되어 AI 가 자동으로 해줍니다.**
2. 🔔 **원본 봇 (`blumn-plan/blumnAI-qa-bot`) 에 새 버전이 나오면** 팀 봇 화면 상단에 배너가 자동으로 표시됩니다 → 🟠 업데이트하기 프롬프트를 한 번 실행하면 30초 만에 갱신 (팀 정책 파일은 절대로 건드리지 않아요)
3. 🔄 **팀 정책 변경은 봇 화면 안에서** 이루어집니다 — 사용자가 [📤 기획전달] 을 누르면 AI 가 정책 md 를 자동으로 수정
4. 🎁 **원본 봇 자체에 개선 아이디어가 있으면** 봇 툴바의 [🎁] 클릭 → GitHub Issue 가 자동으로 생성됩니다. 팀 사본을 수정해도 원본에는 자동으로 반영되지 않으니, 반드시 PR + 리뷰 절차를 거쳐야 합니다 ([CONTRIBUTING.md](CONTRIBUTING.md) 참고)

자세한 원본-사본 원칙은 [docs/05-UPGRADE.md](docs/05-UPGRADE.md) 에 정리해뒀습니다.

## 🟢 세팅하기 — 처음 깔 때 (필수)

**클로드코드 창을 옆에 두고 프롬프트 하나만 붙여넣으면 됩니다** (설치 15-30분).

### 1. 미리 준비할 3가지

| # | 항목 | 어떻게 | 소요 시간 |
|---|---|---|---|
| 1 | **Anthropic API key** — 봇 답변 엔진 | IT기획팀 지라에 요청 ([C04-3284](https://blumnai.atlassian.net/browse/C04-3284) 양식 참고). ⚠️ Claude Code Max 구독과는 별개의 계정입니다 | 반나절 ~ 1일 대기 |
| 2 | **Cloudflare 계정** — Worker 무료 호스팅 | https://dash.cloudflare.com/sign-up 에서 이메일 + 비밀번호로 가입 → 인증 메일 클릭 → 카드 등록 화면은 **Skip** (무료 tier 라 카드 불필요) | 3분 (👤 본인이 직접) |
| 3 | **Node.js 18+** | 미리 확인 안 해도 OK — AI 가 첫 스텝에서 자동으로 확인해줍니다 | 없으면 +5분 |

**전체 준비 소요 시간**: 대부분 IT기획팀 응답을 기다리는 시간입니다.

### 2. 팀 정보 정리

프롬프트에 채워 넣을 값들입니다. **정확히 몰라도 됩니다** — `[모름]` 이라고 적으면 AI 가 GitHub 을 자동으로 검색해서 후보를 찾아 확인 요청해줍니다:

- 팀 이름 (필수)
- GitHub 조직명 (예: `blumn`)
- 정책·화면설계서가 있는 레포 (예: `blumn/○○-planer`)
- 프로젝트별 정보 (프로젝트 = 하나의 서비스. 여러 개 가능. 예: `admin_v1`, `backoffice_v2`)
  - 프로젝트 ID (영문 소문자)
  - 정책 md 폴더 경로
  - 화면설계서 md 폴더 경로
  - 서비스 코드 레포 (선택)

> 🔒 **시크릿 값 (`ghp_...`, `sk-ant-...`) 을 지금 손에 들고 있을 필요는 없습니다.** 셋업 중에 AI 가 발급 화면을 안내해주고, 그 자리에서 터미널 프롬프트에 직접 붙여넣으면 됩니다. **AI 채팅창에는 절대 붙여넣지 마세요.**

### 1. 커뮤니케이션 비용의 극적 절감
- **논리적 설득 자동화** — QA·개발자와 기획자 간의 불필요한 말싸움이 사라짐. AI 가 변경 전/후·근거·대상 파일 위치까지 정제된 논리로 기획자 설득
- **지라 티켓 작성 생략** — 시스템이 명확한 변경 요청을 `qa/decisions/` 에 md 로 자동 저장

### 2. 기획자 리소스 파괴적 절감
- **복사–붙여넣기–배포** — 기획자가 문서 직접 안 열고, AI 가 만든 프롬프트를 Claude Code 에 붙여넣기만 하면 정책·답변 룰·화면설계서 동시 자동 업데이트
- **빠른 이슈 파악** — 상단 요약본 3초 확인 → 배포

### 3. AI-to-AI 소통으로 왜곡 X
사람 개입 시 발생하는 컨텍스트 오해·휴먼 에러 없음. 답변 AI → 기획단 AI 직접 전달.

### 4. 개발 레이어까지 확장
업데이트 내용이 개발자에게도 프롬프트 형태로 자동 전달 → 코드·환경에 바로 적용 가능. 기획-개발 싱크 완벽.

<details>
<summary>⚖️ 한계 · 알아두면 좋은 점</summary>

- **답변 근거는 기본적으로 "선택한 문서 1개" + 프로젝트 안의 다른 문서 목록** — 봇은 현재 선택한 문서의 본문을 우선 근거로 답변합니다. 다른 문서들의 목록도 항상 알고 있어서 *"이 케이스는 캠페인 §2-1 도 참고하세요 (좌측에서 선택)"* 처럼 관련 문서를 안내해줄 수 있어요.  
  여러 문서를 정말로 종합해서 답변받고 싶다면, 설정에서 **전체 문서 참고 모드** 를 켤 수 있습니다 (20개 이하 문서를 가진 팀에게 추천). 자세한 사용법은 아래 [🔀 여러 문서를 함께 참고해서 답변받고 싶다면](#-여러-문서를-함께-참고해서-답변받고-싶다면) 절 참고.

- **코드 검증은 옵션 기능** — 기본 봇은 정책 md 만 참고합니다. `code_repo` 를 설정에 연결하면 봇이 실제 서비스 코드도 함께 참고해서 *"정책은 X 인데 코드는 Y 로 되어있다"* 같은 판정도 가능합니다. 셋업 방법은 [docs/06-CONNECT-CODE.md](docs/06-CONNECT-CODE.md) 참고.

- **정책에 없는 내용은 봇도 모릅니다** — 정책이 부실하면 답변도 부실할 수밖에 없어요. 초기 세팅 후에 [📝 답변 규칙] 과 [📤 기획전달] 을 계속 쌓아가면서 봇이 점점 똑똑해집니다.

</details>

<details>
<summary>🔀 여러 문서를 함께 참고해서 답변받고 싶다면</summary>

기본 동작은 **"좌측에서 선택한 문서 1개"** 를 기준으로 답변합니다. 하지만 실무에서는 *"결제 정책과 알림 정책이 서로 얽혀 있는데..."* 같이 여러 문서를 함께 봐야 할 때가 있죠. 세 가지 방법이 있습니다:

### 방법 1. 봇에게 관련 문서를 안내해달라고 요청 (가장 빠름)

문서 하나 선택한 상태에서 이렇게 물어보세요:

> "이 케이스는 결제 정책과 알림 정책 둘 다 관련이 있을 것 같은데, **어떤 문서들을 봐야 하는지 알려주고** 각각의 관련 절도 함께 안내해주세요."

봇은 프로젝트 안의 모든 문서 목록을 이미 알고 있어서, *"결제포인트 §3-1 + 알림설정 §2-2 를 함께 보시면 좋겠어요"* 처럼 안내해줍니다. 그 다음 좌측에서 해당 문서를 선택해서 이어서 물어보면 됩니다.

### 방법 2. 전체 문서 참고 모드 켜기 (실무에서 가장 강력)

`blumnAI-qa-bot.config.yml` 을 이렇게 수정하고 Worker 재배포:

```yaml
bot:
  include_all_docs: true   # 기본값 false — 이걸 true 로
```

이 모드에서는 봇이 프로젝트 안의 **모든 정책 문서 본문** 을 항상 참고하면서 답변합니다. *"정책 A §3 + 정책 B §2 를 함께 봤을 때 이 케이스는..."* 처럼 여러 문서를 자연스럽게 종합해줍니다.

- **추천 조건**: 프로젝트 안 문서가 20개 이하일 때
- **주의**: 첫 질문은 응답이 3-5배 느려질 수 있음 (전체 문서를 함께 전송). 이후 질문은 캐시가 있어서 원래 속도로 돌아옵니다
- **문서가 많을수록 (30개+)** 봇 답변 품질이 오히려 떨어질 수 있음 (컨텍스트가 너무 커져서 핵심을 놓침). 이 경우 방법 1 을 권장

### 방법 3. 여러 문서를 하나로 합쳐 새 문서 만들기 (근본 해결)

자주 함께 봐야 하는 정책이 있다면, 아예 통합 정책 md 를 하나 만드는 것도 좋은 방법입니다. 예: `결제_알림_통합_정책_v0.1.0.md`. 이렇게 하면 다른 팀원들도 나중에 같은 케이스에서 편해져요.

**요약**:
- 가끔씩 필요 → 방법 1 (봇에게 물어봄)
- 자주 필요 → 방법 2 (설정 켬)
- 팀 자산화 → 방법 3 (통합 문서 만듬)
