# 02. 정책문서 작성 규약 (Wire Policies)

봇이 정확하게 답변하려면 정책 markdown 이 일정 규칙을 따라야 합니다. 이 규약은 단순하지만 어기면 답변 품질이 급락하므로 **반드시** 지켜주세요.

## 위치

```
projects/<프로젝트 id>/docs/policies/<화면이름>_v<SemVer>.md
```

예: `projects/ad_v1/docs/policies/대시보드_v0.1.0.md`

`<프로젝트 id>` 는 `blumnAI-qa-bot.config.yml` 의 `projects[].id` 와 일치해야 함.

## 파일명 규칙

| 패턴 | 예시 |
|---|---|
| `<화면이름>_v<major>.<minor>.<patch>.md` | `대시보드_v0.1.0.md` |
| 화면이름에 띄어쓰기 X → `_` 사용 | `캠페인_만들기_v0.1.4.md` |
| 한글 사용 OK | `결제포인트_관리_v0.1.5.md` |

**버전 시작은 `v0.1.0`**, 본문 변경 시 patch bump (`v0.1.0` → `v0.1.1`).

## 본문 구조 — 4가지 필수 요소

### 1) h1 타이틀에 버전 표기

```markdown
# 대시보드 v0.1.0
```

→ 파일명과 동일한 버전. 봇이 인용할 때 사용.

### 2) 절 헤더에 `§` 번호

봇이 답변 끝에 `(§2-1 참고)` 같이 인용하므로, 절마다 번호를 매겨야 클릭 가능한 anchor 로 동작:

```markdown
## §1. 화면 구성

대시보드는 다음 4개 영역으로 구성된다 ...

## §2. 알림 배너 정책

### §2-1. 표시 조건
- 잔액이 한도 미만이면 주황 배너 ...

### §2-2. 닫기 동작
- 주황 배너 — 닫으면 24시간 미표시 ...
```

**규칙**:
- `##` 는 대절(§1, §2 ...), `###` 는 소절(§1-1, §1-2 ...)
- 번호는 1부터 연번, 중간 번호 재사용 금지
- 절 제목 안에 다른 기호(`-`, `[`, `(`) 사용 자제 — anchor 인식 깨질 수 있음

### 3) 짧은 단락 + 불릿

긴 문단보다 짧은 줄이 봇이 인용하기 좋음. 한 절 안에 2-7개 불릿 또는 1-2 단락이 적정.

❌ 나쁜 예 (긴 문단):
```markdown
### §2-1. 표시 조건
잔액이 발송 가능 한도 미만이면 주황 배너가 표시되며, 이때 사용자는 충전하기 버튼을 통해 즉시 충전 페이지로 이동할 수 있고, 외부 연동이 끊겼을 때는 빨강 배너가 표시되어 닫을 수 없으며 ...
```

✅ 좋은 예:
```markdown
### §2-1. 표시 조건

- 잔액이 발송 가능 한도 미만 → 주황 배너 + [충전하기] 버튼
- 외부 연동 끊김 → 빨강 배너 (닫기 불가) + [재연결하기] 버튼
```

### 4) 시각 명세 — 색·문구·버튼 명시

봇이 답변할 때 *"이렇게 노출된다"* 라고 말하려면 정책에 시각 정보가 들어 있어야 함:

✅ 좋은 예:
```markdown
- 주황 배너 + `포인트 충전하면 발송 가능합니다` 문구 + [충전하기] 버튼
- 빨강 배너 + `연동 만료` 문구 + [재연결하기] 버튼 (닫을 수 없음)
```

❌ 나쁜 예:
```markdown
- destructive 타입 인라인 배너로 표시됨
- severity 3 알림 노출
```

## 선택 요소 — 권장하지만 필수 아님

### 버전 이력 표 (h1 바로 아래)

```markdown
# 대시보드 v0.1.2

| 버전 | 일자 | 변경 |
|---|---|---|
| v0.1.2 | 2026-06-15 | §4-1 알림 정렬 규칙 추가 |
| v0.1.1 | 2026-05-29 | §2-2 빨강 배너 닫기 불가 명시 |
| v0.1.0 | 2026-05-20 | 초기 |
```

→ 봇이 답변 시 "최근 변경" 컨텍스트로 활용.

### 📍 변경 위치 빠른 색인 (대형 정책문서일 때)

```markdown
## 📍 v0.1.2 변경 위치 빠른 색인

| 변경 | 위치 |
|---|---|
| 알림 정렬: 심각도 내림차순 | §4-1 |
| 빨강 배너 닫기 불가 명시 | §2-2 |
```

→ 협업자가 "뭐가 바뀌었어요?" 물을 때 봇이 즉답 가능.

## 정책 변경 절차 — 2가지 규칙

### A) 작은 변경 (patch bump)

오타·표현 다듬기·작은 행동 변경:
1. 본문 직접 수정
2. 파일명 `_v0.1.0.md` → `_v0.1.1.md` rename
3. h1 타이틀 버전 갱신
4. 버전 이력 표에 행 1줄 추가

### B) 의미 변경 (minor bump)

새 절 신설·검증 룰 변경·필드 추가:
1. 구버전 파일을 `_old/` 폴더에 박제 (`_old/대시보드_v0.1.5.md`)
2. 원본 파일명 `_v0.1.5.md` → `_v0.2.0.md`
3. cross-link 갱신 — 다른 정책·workflow 에서 `_v0.1.5` 참조하는 곳 모두 새 버전으로

이 절차는 헤이데어 기획팀의 정책 버전관리 규칙 §4 를 기준으로 정리한 것. 팀 자체 규칙이 있으면 그것으로 바꿔 운영해도 OK — 봇은 파일명·헤더 형식만 지키면 되므로 세부 절차는 자유.

## 화면설계서(storyboard) — 정책과 다른 폴더

storyboard 가 있다면 별도 폴더:
```
projects/<프로젝트 id>/docs/storyboards/<화면이름>/<화면이름>_storyboard_v0.1.0.md
projects/<프로젝트 id>/docs/storyboards/<화면이름>/<화면이름>_storyboard_v0.1.0.html
```

→ md 옆에 같은 이름 html 두면 봇이 "미리보기" 모드로 보여줌.

`config.yml` 의 `projects[].storyboards_dir` 와 `storyboard_layout` 설정 필요.

## 좋은 예 — 전체 한 문서

```markdown
# 대시보드 v0.1.2

| 버전 | 일자 | 변경 |
|---|---|---|
| v0.1.2 | 2026-06-15 | §4-1 알림 정렬 규칙 추가 |
| v0.1.1 | 2026-05-29 | §2-2 빨강 배너 닫기 불가 명시 |
| v0.1.0 | 2026-05-20 | 초기 |

## §1. 화면 구성

대시보드는 다음 4개 영역으로 구성된다:

1. 상단 알림 배너
2. 핵심 지표 카드 (좌)
3. 시계열 차트 (우)
4. 최근 활동 리스트 (하단)

## §2. 알림 배너 정책

### §2-1. 표시 조건

- 잔액 < 발송 한도 → 주황 배너 + `포인트 충전하면 발송 가능합니다` 문구 + [충전하기] 버튼
- 외부 연동 끊김 → 빨강 배너 + `연동 만료` 문구 + [재연결하기] 버튼

### §2-2. 닫기 동작

- 주황 배너 — 사용자가 닫으면 24시간 동안 미표시
- 빨강 배너 — 닫기 버튼 없음 (재연결 전까지 계속 노출)

## §4. 알림 드롭다운 (미확인 탭)

### §4-1. 정렬

심각도 내림차순 (3 → 2 → 1). 동일 심각도면 최신순.
```

## 잘 작성됐는지 검증 체크리스트

새 정책 만들 때마다 확인:

- [ ] 파일명에 `_v0.X.Y` 버전 표기
- [ ] h1 타이틀에 동일 버전
- [ ] 모든 `##`·`###` 헤더에 `§번호` 표기
- [ ] 각 절 본문이 7줄 이내, 불릿 위주
- [ ] 시각 명세(색·문구·버튼) 들어 있음
- [ ] 봇에 "이 정책 §2-1 요약해줘" 물어봤을 때 만족스러운 답변

마지막 항목이 가장 중요. 봇 답변이 이상하면 정책 본문에 정보가 빠진 것입니다.

## 🩺 자동 검증 CLI

체크리스트 위 5가지 항목 (파일명·h1·§번호·본문 길이·시각 명세) 을 **자동으로 스캔** 하는 CLI 도구가 코어에 포함되어 있어요:

```bash
# 팀 정책 레포 루트에서 실행 (모든 projects/*/docs/policies 자동 감지)
node .blumnAI-qa-bot/scripts/check-policies.mjs

# 또는 특정 폴더 지정
node .blumnAI-qa-bot/scripts/check-policies.mjs projects/admin_v1/docs/policies
```

**출력 예시**:

```
📊 정책 md 규약 검증 리포트
  스캔 폴더: 1개
  대상 파일: 12개
  실패: 2건 · 경고: 3건

  ✅ projects/admin_v1/docs/policies/대시보드_v0.1.5.md
  ❌ projects/admin_v1/docs/policies/캠페인_v0.1.md
      · ❌ 파일명 규약 위반 — <이름>_v<X.Y.Z>.md 형식이어야 함
  ⚠️  projects/admin_v1/docs/policies/설정_v0.1.0.md
      · ⚠️  ## §3. 알림 — 본문 15줄 (권장 ≤ 12줄, 봇 답변 품질 저하)
      · ⚠️  시각 명세 (색·문구·버튼) 힌트 감지 안 됨
```

- **exit code 0**: 모두 통과
- **exit code 1**: 실패(❌) 있음 → CI 에서 fail 처리 가능
- **exit code 2**: 스캔 대상 없음

**🔵 점검하기 프롬프트** 안에서 AI 가 이 CLI 를 자동 실행하고 리포트를 해석해줍니다 — 사용자가 직접 실행할 필요 X. 다만 CI 에 훅 걸어두면 정책 md PR 마다 자동 검증되어 편해요.
