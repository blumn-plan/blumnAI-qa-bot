## ⚡ Quick Start — 프롬프트로 자동 진행

혼자 시나리오 5개를 한 번에 훑어보고 싶으면 Claude Code 창에:

```
방금 blumnAI-qa-bot 설치 완료했습니다. docs/07-FIRST-TEST.md 의
시나리오 5개를 순서대로 안내해주세요.

각 시나리오마다:
1. "이 질문을 봇 창에 던져보세요" — 정확한 질문 문구 제시
2. "예상 답변 형태" — 어떤 형식·요소가 답에 있으면 정상
3. 제가 결과를 붙여넣으면 → 정상/이상 판정 + 다음 시나리오 or 트러블슈팅
```

Claude Code 가 아래 시나리오를 하나씩 안내해줍니다.

## 시나리오 1 — "정책 직접 인용" (기본 답변)

**목적**: 봇이 정책 md 를 근거로 답하는지 확인.

**준비**: 좌측 정책문서 목록에서 아무거나 하나 클릭 → 가운데 영역에 md 가 렌더링됨.

**질문 예시** (본인 팀 정책에 맞게 각색):
```
이 문서 §1 은 뭘 정의하고 있어? 3줄로 요약.
```

**정상 답변 형태**:
- 결론 한 줄로 시작
- 전체 답 7줄 이내
- 답 끝에 `(§1 참고)` 같은 § 인용
- § 인용 클릭하면 가운데 영역이 해당 절로 자동 스크롤 + 노란 하이라이트

**이상하면**:
- 답이 "정책 미정의" → 정책 md 에 §번호 표기 누락 가능성 → [02-WIRE-POLICIES §본문 구조](02-WIRE-POLICIES.md#본문-구조--4가지-필수-요소)
- § 클릭이 안 됨 → h1 타이틀에 버전이 없거나 헤더 형식 안 맞음
- 답이 너무 김 → answer-rules.md 의 §A-0 규칙이 세지 않음

---

## 시나리오 2 — "정책 → 화면 매핑" (시각 명세)

**목적**: 정책의 시각 명세(색·문구·버튼)를 봇이 답에 반영하는지.

**질문 예시**:
```
[화면 이름] 에서 [특정 상태] 일 때 사용자에게 뭐가 어떻게 보여?
색깔·문구·버튼 다 알려줘.
```

예: `대시보드에서 잔액이 부족할 때 사용자에게 뭐가 어떻게 보여? 색깔·문구·버튼 다 알려줘.`

**정상 답변 형태**:
- "주황 배너 + `포인트 충전하세요` 문구 + [충전하기] 버튼" 같이 **시각적으로 묘사됨**
- 정책 md 안의 시각 명세를 그대로 인용

**이상하면**:
- "포인트 부족 알림이 표시됩니다" 같이 두루뭉술 → 정책 md 에 시각 명세 부실. [02-WIRE-POLICIES §시각 명세](02-WIRE-POLICIES.md#4-시각-명세--색문구버튼-명시)
- 답이 답인데 색·문구 다 다름 → 정책 md 를 봇이 못 읽고 있을 가능성. 좌측에서 문서 선택 여부 확인

---

## 시나리오 3 — "화면 캡처 첨부" (drift 감지)

**목적**: 사용자가 화면 캡처를 붙였을 때 정책 vs 화면 drift 를 봇이 판정하는지.

**준비**: 본인 서비스의 실제 화면 캡처 하나 준비. 좌측에서 관련 정책 md 를 이미 선택해둔 상태로.

**방법**:
1. 채팅 입력창에 캡처 이미지를 **Ctrl+V 붙여넣기** (또는 드래그 앤 드롭)
2. 텍스트로 함께:
```
정책과 실제 화면이 다른 것 같은데 어느 쪽이 맞아? 차이가 있으면 표로 정리해줘.
```

**정상 답변 형태**:
- 이미지 첨부됐으니 봇이 시각 분석
- 차이 있으면: `정책상 X 이지만 첨부 화면은 Y. 정책 또는 화면 보완 필요.` 표기
- 차이 없으면: "정책과 화면이 일치합니다" 로 마무리
- 판정 어려우면 어느 부분이 애매한지 명시

**이상하면**:
- 이미지가 그냥 무시됨 → CLAUDE_MODEL 이 vision 미지원 모델 (config `bot.claude_model` 확인)
- 항상 "일치" 라고만 답 → answer-rules.md 의 §A-2 (drift) 규칙 누락

---

## 시나리오 4 — "코드 검증" (코드 연결한 팀만)

**목적**: `code_repo` 를 연결한 팀이라면, 정책 vs 코드 drift 판정이 실제로 작동하는지.

> [06-CONNECT-CODE.md](06-CONNECT-CODE.md) 를 완료한 팀만. 안 했으면 시나리오 5로 스킵.

**질문 예시**:
```
[특정 기능] 은 정책상 어떻게 되고, 실제 코드는 어디에서 처리하고 있어?
정책과 코드가 다르면 알려줘.
```

예: `캠페인 삭제는 정책상 어떻게 되고, 실제 코드는 어디에서 처리하고 있어? 정책과 코드가 다르면 알려줘.`

**정상 답변 형태**:
- 답 안에 `src/pages/○○/○○.tsx:42` 같은 **파일 경로 + 라인 번호** 등장
- 정책 인용 (§X-Y) + 코드 인용이 함께
- 다르면: `정책상 X 이지만 코드(파일:라인) 는 Y. 정책 또는 코드 수정 필요.` 표기

**이상하면**:
- 코드 언급이 아예 없음 → `config.yml` 의 `code_repo` 미채움 또는 PAT scope 문제 → [06-CONNECT-CODE §트러블슈팅](06-CONNECT-CODE.md#트러블슈팅)
- 답 나오는 데 20초+ → GitHub Search rate limit 근처. `code_max_snippets` 를 낮추기
- 파일 경로가 있는데 실제로는 없는 파일 → `answer-rules.md` §A-3 (코드 인용 규칙) 강화 필요

## 시나리오 5 — "개선 요청 → 재답변" (feedback 루프)

**목적**: 답변 톤·형식이 불편할 때 개선 요청이 즉시 반영되는지.

**준비**: 시나리오 1의 답변이 화면에 남아 있는 상태.

**방법**:
1. 그 답변 카드의 [📝 **개선 요청**] 버튼 클릭
2. 개선 요청 사항 입력 (예: `기술용어 빼고 3줄 이내로 요약`)
3. ☑ **저장 후 이 답변 다시 받기** 체크 → [저장]

**정상 동작**:
- 즉시 같은 질문이 재실행됨 (🔁 태그 + 초록 테두리)
- 새 답변이 요청한 대로 짧아지거나 톤이 바뀜
- 왼쪽 사이드바 **📝 AI봇 답변 개선건** 에 방금 저장된 룰 등장
- `qa/feedback/YYYY-MM-DD-...md` 파일이 GitHub 에 자동 commit

**이상하면**:
- 저장 후 재답변이 안 옴 → 워커 배포 실패 or PAT write 권한 문제. F12 → Network 탭 확인
- 파일이 GitHub 에 안 올라감 → PAT 이 read-only. `repo` scope 재확인
- 재답변이 개선 요청 무시 → answer-rules.md 가 `.blumnAI-qa-bot/answer-rules.md` 경로에 있는지 확인
